Every routed change is an experiment. One row per change. The orchestrator
scores rows at every retro (Â§26). Statuses: `active` | `validated` |
`validated-by-null-hypothesis` | `under-question` | `retirement-trial` |

**THE VALIDITY BAR â€” every row MUST be a falsifiable HYPOTHESIS, not a piece of
work (EXP-063).** A row is admissible only if it states all four, explicitly and
checkably:
1. **Problem** â€” the specific evidenced friction/gap (from the ledger /
   principle-failures / a defect). Not "a thing we want to build".
2. **Solution** â€” the concrete change made (the intervention being tested).
3. **Target DORA metric** â€” a NAMED metric it aims to move: lead time /
   deployment frequency / CFR / MTTR. A meta/proxy metric (agent-context-size,
   registry-validity) is allowed ONLY when the row explicitly justifies it as a
   DORA proxy.
4. **Measurement** â€” how we will SEE if it helped: the observable signal + a
   scoring horizon, phrased so the result CAN come back NEGATIVE.

**REJECT on sight (delete the row) any row that is really:** a feature /
capability / "piece of work to do" description; OR has no named target DORA
metric; OR has a "measurement" that cannot fail (a did-we-do-the-work checklist
â€” "the documenter produces consumer docs", "the architect states fitness
functions" â€” is NOT a falsifiable measurement). Lifecycle is adopt-or-delete:
run enough trials â†’ if the metric moved, ADOPT (fold the behaviour into the
owning agent as plain practice, prune the row to the archive); if it did NOT,
UNDO the change and DELETE the row. A sound, load-bearing shipped behaviour whose
row was merely mis-phrased as a work-item is handled by DELETING the row while
KEEPING the behaviour as plain agent practice â€” never undo a behaviour that
prevents a known defect class just because its row failed the bar.

**This file holds ONLY LIVE experiments.** When an experiment reaches a terminal
state â€” `integrated` (behaviour folded into its agent file as plain practice),
`retired`, or `reworkedâ†’EXP-NNN` â€” its row is **PRUNED from here** and replaced
by one terse line in `process/experiments-archive.md` (the index of what we've
learned and folded in). The full row stays recoverable in git. **A `failed`
experiment (anticipated effect not observed AND abandoned/superseded) is DELETED
OUTRIGHT â€” no archive line (Â§25a.6)** â€” because failed rows are the most verbose
and carry no live scoring thread, so they only pollute the registry; any durable
lesson is carried by a successor experiment or a `principle-failures/` note, and
git retains the full row. Failed rows may be deleted as soon as recognised, not
only at a retro. The registry must stay small: each retro prunes the rows that
reached terminal state (v45 Â§25a).

**Horizon** counts SCORING OPPORTUNITIES (slices/iterations where the change
could have shown its effect), not calendar retros. "No opportunity" extends.
**Retirement-trials run 4â€“5 scoring opportunities** before "retired" may be
concluded; an attributable metric drop reinstates early.
**Validated rows trigger INTEGRATION (v34):** the owning agent file is rewritten
so the behaviour becomes core operating practice (scaffolding removed, mechanism
intact); then the row is pruned to the archive (v45). EXP-011 scores the
integration+pruning policy itself.

**Applies-to (v38):** each experiment names the KIND of work that exercises it
(in its row's scoring notes as `applies-to: ...`). An experiment is a scoring
opportunity ONLY for matching work. At work selection the orchestrator lists the
applicable active experiments (Â§10/Â§25a).
