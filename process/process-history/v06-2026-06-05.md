# Current Process — v6 (archived)

The process all agents follow right now. Updated only by the Orchestrator at a
retro, which snapshots the prior version into `process-history/` first.

## 1. Operating principles (beliefs)

See `principles/` for the full statements. In force: XP, always-TDD, value
slicing, trunk-based development, continuous deployment, roll-forward-with-
reversible-rollback, defect-as-spec, jobs-to-be-done. Treat these as defaults,
not laws — deviations are allowed but must be logged in `principle-failures/`.

## 2. Gross lead time — correct definition

**Gross lead time = wall-clock time from idea accepted → running in prod.**
It includes ALL time: agent processing, human gate approval waits, session idle
time, overnight pauses. Everything. If the system is sitting waiting for a human
who is asleep, that wait is part of lead time.

From ox (2 slices measured):
- Total wall clock: ~24.5 hours
- Agent processing: ~2.1 hours (9%)
- Wait / idle time: ~22.4 hours (91%)

**Agent speed is not the primary lever. Gate count and gate design are.**

The DORA script must compute this as:
`deploy.timestamp − slice_accepted_gate.timestamp`
for each slice. The orchestrator must ensure both events are in the ledger.

## 3. Loops

- **New requirement** → `/requirement-new`: JTBD vision → target architecture →
  Chunks → required capabilities. Gated.
- **Per iteration** → `/iteration-run`: next slice → arch delta → acceptance
  tests → thin route → TDD build → test in prod → **document** → **retro**
  (both mandatory; run automatically at slice delivery without human trigger).
- **Retro** → `/retro`: recompute DORA, review principle-failures, answer the
  focus question, propose + apply a process change. Runs after every slice.
  One data point is never sufficient to change a principle — require a pattern.

## 4. Gate design (the primary lead-time lever)

Each human gate is a potential overnight wait. The system minimises blocking
gates by:

**a. Auto-approve where the outcome is clear:**
- Go/no-go to deploy: orchestrator auto-approves (and dispatches tester
  immediately) when — project is local-only AND all tests pass AND engineer
  reports no principle deviations. Human attention not required.
- Future: consider whether arch + security review for local-only projects with
  no new infrastructure and no security-surface changes can be self-certified by
  the architect without human sign-off.

**b. Never block on a single gate when independent work can proceed:**
- As soon as slice N's gate 2B is approved (slice accepted), immediately
  dispatch product + architect for slice N+1 in parallel with the engineer
  building slice N. When the human returns to approve N+1's gates, work can
  start without another agent dispatch cycle.
- This turns gate wait time into productive planning time.

**c. Batch gate decisions where possible:**
- When multiple gate decisions are pending (e.g. the human was away and both
  the arch review and the slice definition completed), surface them together in
  a single prompt rather than sequentially. The human approves once and
  everything unblocks.

**Anticipated DORA effect:** each gate removed or auto-approved from the
critical path can save multiple hours of wall-clock lead time. Observable as
reduction in gross lead time (wall-clock) per slice from slice 003 onward.

## 5. Project classification (solution-architect guidance)

At the start of the architecture phase, classify the project:

- **Cloud/hosted** (has networking, persistence, multi-user, or a deployed
  service): apply the full AWS Well-Architected default — account & network
  structure, IAM, use `aws-architecture` skill.
- **Local-only** (CLI tool, library, script — no network, no persistence, no
  hosted service): skip cloud scaffolding. Apply Well-Architected Security,
  Reliability, and Operational Excellence pillars at source/process scope.
  Record the justified deviation once in `principle-failures/`; do not repeat
  it per slice.

## 7. Tester scope

The tester validates **customer-observable outcomes** through the most
public-facing surface. It does NOT re-implement algorithmic or exhaustive
correctness checks owned by the engineer's test suite.

Tester responsibilities:
- Run `pytest` (or equivalent) and confirm the suite passes.
- Exercise the public surface behaviourally: happy path, key error cases,
  adversarial UX inputs.
- Validate the success measures from `slice.md` as a user would experience them.

Note: tester agent speed (1650s median) is secondary to gate latency in the
lead time picture. Both matter, but gate count matters more.

**Anticipated DORA effect on agent work time:** tester median < 1200s.
Observable at slice 003.

## 8. Deploy event logging

The orchestrator logs a `deploy` event row immediately when the tester passes,
before dispatching the documenter. The `slice_accepted` gate event (gate 2B)
is the lead time start. The deploy event is the lead time end.

Orchestrator also logs a `slice_accepted` event at gate 2B (or equivalent) so
the DORA script can compute wall-clock gross lead time as:
`deploy.timestamp − slice_accepted.timestamp`

The orchestrator updates `work/<project>/dora/per-project.md` at the end of
each slice retro. Not delegated to the engineer.

## 12. Deploy event — definition by project type

**What counts as a deploy** determines when the `deploy` ledger event is logged
and therefore what gross lead time measures. It must be the point at which
the change is available to real users.

| Project type | Deploy event trigger | Logged by |
|---|---|---|
| Cloud/hosted (AWS, Azure) | CI/CD pipeline completes and change is live in production (not staging) | cicd agent or engineer on pipeline success |
| Local CLI / library (e.g. ox) | Tester validation passes — there is no infra deploy step; the runnable artefact is the product | orchestrator, immediately after tester task_end |

**The ox exception:** ox is a local console application. The "deploy" is the
tester confirming it works correctly — there is no AWS stack, no pipeline, no
container to push. This is not the normal case. Most real projects will have a
distinct deploy step (ECR push, ECS service update, Lambda publish, etc.) that
must complete before the tester validates the live environment.

For cloud/hosted projects: do NOT log the deploy event at tester-pass. Log it
when the pipeline confirms the change is running in production. The tester then
validates against that live system.

**Why this matters for DORA:** gross lead time = idea → running in production
for real users. Conflating deploy with test-pass on a local tool is acceptable
for a toy project but would undercount lead time on a real system where infra
propagation, blue/green cutover, or cache warming add significant time.

## 14. Change-set queued for next iteration

**Engineer duration_s (v6 — data gap fix):**
The engineer must populate `duration_s` in its task_end ledger row with
wall-clock seconds from task_start to task_end. Without it the DORA baseline
has no build-step data and the constraint model is incomplete.

**Anticipated DORA effect:** engineer appears in baseline with real data;
constraint model covers all 4 measured agent steps.

**Deploy event format fix (v6 — measurement gap):**
`dora.py compute` reports 0 deploys despite deploy rows in the ledger.
Investigate and fix the event row format or the script so gross lead time
(wall-clock, slice_accepted → deploy) becomes computable. Orchestrator to
verify the script produces non-zero gross lead time after the first slice of
the next project.

**Anticipated DORA effect:** gross lead time and deployment frequency become
real numbers; retro focus question answerable from data, not manual arithmetic.
