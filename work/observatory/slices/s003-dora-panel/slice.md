# s003 — CHK-3 DORA panel: four metrics + stage cards + time-thief view

## Job served

**[CORE] Observe DORA and flow metrics (J1)**

When the operator wants to know whether the pipeline is healthy and where the
constraint sits, they want to see the four DORA metrics, per-agent task times,
and the biggest time thieves — all at a glance and traceable to source — so they
can act on real numbers rather than intuition.

Killick test: YES — after this slice the operator can read every metric from
`baseline.md` and `flow.md` on-screen with one click back to source, replacing
manual `cat process/dora/baseline.md` + `cat work/<project>/dora/flow.md`.

## Architecture note — markdown parse question

**Decision: client-side parse (recommended lighter path).**

The CHK-1 read layer (`UC3 / s001`) already serves `baseline.md` and `flow.md`
as raw markdown strings via:
- `GET /api/dora/baseline` → `{ content: string | null }`
- `GET /api/projects/:id/dora/flow` → `{ content: string | null }`

Both formats are **stable and well-known** (markdown tables, consistent headers,
generated by `dora.py compute` and `dora.py flow`). Parsing on the client is an
S-cost TypeScript regex/table-parse against known column names — no server change
needed, no new endpoint, no added deployment risk.

The alternative (a `/api/dora/baseline/parsed` endpoint on the server) would be
an M-cost backend change that splits the parse contract across layers and bloats
the server's scope. The s001 server contract is intentional raw pass-through.

**Architect/cicd gate:** confirm client-side parse OR approve a server-side parse
endpoint. The UC1 parser seam in this slice isolates that decision: if the
architect opts for server-side parse, UC1 changes its input source but its output
shape (the typed records) and all downstream UCs remain unchanged.

The s002 SPA already has `src/app/parsers/baseline.ts` (constraint extraction).
UC1 of this slice **extends** that parser module with the full metric extraction.

## Scope: what IS in this slice

1. **Markdown parsers** — extend `src/app/parsers/baseline.ts` to extract:
   (a) four DORA metrics + windows from the `## Four key metrics` table;
   (b) per-agent task times (modal/median/mean) from the `## Per-agent task
   completion` table; (c) ToC constraint name (already exists in s002 — reuse).
   Extend or add `src/app/parsers/flow.ts` to extract: (a) time-thief rows from
   the `## Time thieves` table; (b) queue throughput/dwell/rework from the
   `## Queues` table. Each parser function returns typed records or `null` on
   absent/unparseable input (fail soft, no crash).

2. **DORA four-metric panel** — a new panel component rendering the four DORA
   metrics (gross lead time, deployment frequency, change failure rate, MTTR) with
   their windows, sourced from `baseline.md`. Each figure links to its source file
   (`process/dora/baseline.md`, highlighted row). Empty-state when `baseline.md`
   is absent or metrics are not yet computed.

3. **Stage cards** — per-agent task-time cards (modal/median/mean) sourced from
   `baseline.md` per-agent table. The constraint agent is highlighted (reuse the
   constraint value from the UC5/s002 parser). Each card links to source.

4. **Time-thief view** — ranked list of time thieves from `flow.md`
   `## Time thieves` table (dwell, collisions, parallelism efficiency). Each row
   links to source. Empty-state when `flow.md` is absent.

5. **Source-link traceability** — every rendered figure has a `data-source`
   attribute (or visible link) carrying the source file path and the section
   heading (e.g. `process/dora/baseline.md#four-key-metrics`). The traceability
   requirement (§8 NFR + SM3) means each number traces back to its exact row.

6. **SSE live refresh** — reuse the existing SSE seam from s002 (`src/app/sse/`).
   A `change` event for `process/dora/baseline.md` or `work/<project>/dora/flow.md`
   triggers a re-fetch and re-parse of the affected panel. No new SSE infrastructure.

## Scope: what is explicitly NOT in this slice

- **Trend derivation from `ledger.csv`** — requirements §7 says "trend where
  derivable from ledger.csv"; this is a nice-to-have extension, not core for CHK-3
  done-condition. Deferred to a later improvement slice.
- **Queue buffer state on stage cards** — queue lengths/flags remain on the pipeline
  map (s002). Stage cards show task times only.
- **Per-item lead time table** — the `## Per-item lead time` section of `flow.md`
  is a long detail table; not rendered in this slice (CHK-4 drill-down territory).
- **Drill-down into individual agent tasks / ledger history** — CHK-4.
- **Collision detail rows** — the collision table in `flow.md` is visible in the
  time-thief view (the "hidden-edge collisions" thief row links to flow.md source)
  but the individual collision rows are not separately rendered here.
- **Stage card queue throughput / dwell** — the `## Queues` table in `flow.md`
  includes per-queue throughput and dwell; only the time-thief ranking is rendered
  in this slice. Full queue detail is CHK-4 / future.
- **Any write/steer affordance** — Phase 1 is strictly read-only.
- **Mobile/responsive layout** — out of scope for v1.
